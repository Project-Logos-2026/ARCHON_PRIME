## Category 1: `venv/` — Python Virtual Environment (863 files)

### What it is
A Python 3.12 virtual environment created manually during the Codespaces recovery session when the container was running in Alpine recovery mode. It was created with:
```
python3 -m venv /workspaces/ARCHON_PRIME/venv
```

### Why it exists
During the recovery-mode session, the correct devcontainer image (`mcr.microsoft.com/devcontainers/python:3.12`) was not running. A manual venv was created to attempt local package installation. After the container was rebuilt correctly, this `venv/` directory persisted in the workspace filesystem.

### Contents breakdown

| Subcategory | File Count | Description |
|---|---|---|
| `venv/lib/site-packages/pip` | 842 | pip 25.0.1 package source files (`.py` modules) |
| `venv/lib/site-packages/pip-25.0.1.dist-info` | 9 | pip distribution metadata |
| `venv/bin` | 10 | Activation scripts + python/pip symlinks |
| `venv/pyvenv.cfg` | 1 | Virtual environment config |
| Other `venv/lib/` | 1 | `__pycache__` / base lib |

### Key details
- **Python version:** 3.12.12
- **Only package installed:** `pip 25.0.1` — the `requirements.txt` packages (`bandit`, `black`, `GitPython`, etc.) were **NOT** installed into this venv
- **`pyvenv.cfg`:** `include-system-site-packages = false` — isolated from system packages
- **All python/pip binaries are symlinks** to `/usr/local/bin/python3.12` (the system Python)
- **This venv is NOT active** — `$VIRTUAL_ENV` is empty in the current terminal
- **Not referenced** in `devcontainer.json` or `Machine/settings.json`
- **Referenced incorrectly** as `.venv` (not `venv`) in `.vscode/settings.json`

### VS Code Python extension impact
`.vscode/settings.json` points to `${workspaceFolder}/.venv/bin/python` (with a leading dot). This path **does not exist**. The manually created `venv/` (without dot) is therefore:
- Not selected by the Python extension
- Not auto-activated in terminals (despite `python.terminal.activateEnvironment: true`)
- Generating a "Python interpreter not found" warning on every session start

### Remediation status: ⚠️ NEEDS ACTION
This directory should either be:
1. **Deleted** (recommended) — the system Python at `/usr/local/bin/python` with all packages pre-installed in system site-packages is what was configured by `devcontainer.json` and `postCreateCommand`, and it is fully functional
2. **Kept and `.gitignore`d** — if a project-specific venv is desired, the path in `.vscode/settings.json` must be corrected from `.venv` → `venv`

## Recommended Actions (in order of priority)

### 1. Fix `.vscode/settings.json` interpreter path
**File:** [`.vscode/settings.json`](.vscode/settings.json)  
**Change:** `"python.defaultInterpreterPath": "${workspaceFolder}/.venv/bin/python"`  
→ `"python.defaultInterpreterPath": "/usr/local/bin/python"`

This eliminates the Python extension warning and aligns with both `devcontainer.json` and Machine settings.

### 2. Add a `.gitignore`
Create `.gitignore` at repo root to suppress `venv/`, `__pycache__`, and other generated artifacts from polluting source control:
```gitignore
venv/
.venv/
__pycache__/
*.pyc
*.pyo
.pytest_cache/
.mypy_cache/
.ruff_cache/
coverage.xml
*.egg-info/
```
This alone will reduce the "875 changes" to 12 (just the audit artifacts).

### 3. Decide on `venv/`
Either:
- **Delete it:** `rm -rf venv/` — the system Python is complete and correct
- **Keep it committed** if a project-local isolated environment is intended (add full `requirements.txt` install inside it)

### 4. Fix Accessible View terminal redirect
In VS Code Settings (`Ctrl+,`), search for:  
`terminal.integrated.accessibleViewFocusOnCommandExecution`  
Set to **`false`**.
